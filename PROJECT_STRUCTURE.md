### Auth Flow
1. `POST /api/auth/login` в†’ Supabase session cookie
2. `middleware.ts` validates cookie on every protected route
3. `AuthContext.tsx` (client) derives role-based permission flags from `profiles` table
4. API routes call `getCallerRole()` from `src/lib/api-auth.ts` for server-side checks

### Data Write Pattern
- **Reads**: `createClient()` (server) вЂ” respects RLS, uses anon key
- **Writes**: `createAdminClient()` (server) вЂ” bypasses RLS, uses service role key
- All write endpoints validate with Zod before calling admin client

### Tables вЂ” Legacy System

Linked to `auth.users`. Used for login/session/role management.
`id`, `full_name`, `role` (system_admin|roster_planner|manager|employee), `created_at`

#### `shifts` / `rosters`
Legacy shift templates and assignments. Kept for backward compatibility.
See `supabase/schema.sql` for full definitions.

### RLS Pattern (all tables)
```sql
-- Authenticated users can read
CREATE POLICY "..." ON <table> FOR SELECT TO authenticated USING (true);
-- Service role bypasses RLS for writes (used by createAdminClient())
CREATE POLICY "..." ON <table> FOR ALL USING (auth.role() = 'service_role');
```

## Component Reference

All components live in `rostering_soft_demo/src/components/`.

### `DataTable<T>`
Generic sortable table with loading and empty states.
```ts
interface Props<T> {
  columns: { key: string; header: string; render?: (item: T) => ReactNode }[];
  data: T[];
  loading?: boolean;
  onEdit?: (item: T) => void;
  onDelete?: (item: T) => void;

```

### `Modal`
Accessible overlay with backdrop click + Escape key to close.
```ts
interface Props {
  open: boolean;
  onClose: () => void;
  title: string;
  children: ReactNode;
}
```

### `FormField`
Label + error wrapper. Exports `Input`, `Select`, `Button` sub-components.
```ts
interface Props { label: string; error?: string; children: ReactNode; }
// Button variants: 'primary' | 'secondary' | 'outline' | 'ghost' | 'danger'
```

### `CascadingFilter`
Department в†’ Roster Group в†’ Designation cascading dropdowns. Shared between `/duties` and `/dispatch`.
```ts
interface Props {
  onFilterChange: (filters: {
    department_id: string;
    roster_group_id: string;
    designation_id: string;

  showDesignation?: boolean;

```

### `Navbar`
Fixed top bar (z-30, h-16). Props: `{ userName?: string }`.

### `Sidebar`
Fixed left nav (z-20, w-64). Grouped sections: Operations / Configuration / Personnel / Scheduling.
Props: `{ role?: UserRole }`. Filters links by role.

## Page Reference

### `/(app)/dashboard`
Shows next shift (legacy `rosters` table) and user role card. Quick links to Rosters and Shifts.

### `/(app)/departments`
CRUD for `departments`. Roles: system_admin, roster_planner.

### `/(app)/designations`
CRUD for `designations`. Roles: system_admin, roster_planner.

### `/(app)/roster-groups`
CRUD for `roster_groups`. Modal has Department + Designation dropdowns + optional end date.

### `/(app)/employees`
CRUD for `employees`. Wide modal (3-col layout). Roster Group dropdown cascades from selected Department.

### `/(app)/duties`
CRUD for `duties`. CascadingFilter bar filters table. Modal: Department в†’ Roster Group в†’ auto-filled Designation. duty_hours computed client-side for preview; stored in DB as generated column.

### `/(app)/dispatch`
Drag-and-drop roster board using `@dnd-kit`. Uses `employees` + `duties` + `duty_assignments` (new system).
- Date range picker (replaces fixed week)
- Department + Roster Group filters
- Employee name / ID search
- Duty pool filtered by active filters
- Drag duty tile в†’ drop on employee+date cell в†’ creates duty_assignment

### `/(app)/shifts` (legacy)
CRUD for legacy `shifts` table.

### `/(app)/rosters` (legacy)
CRUD for legacy `rosters` table (user в†’ shift в†’ date assignments).

### `/(app)/users`
Edit/delete `profiles`. Roles: system_admin, roster_planner (view), system_admin (write).

```

NEXT_PUBLIC_SUPABASE_URL=          # Supabase project URL
NEXT_PUBLIC_SUPABASE_ANON_KEY=     # Public anon key (safe to expose)
SUPABASE_SERVICE_ROLE_KEY=         # Private service role key (server-only)
```

Frontend uses `NEXT_PUBLIC_*` via `@supabase/supabase-js` browser client.
API routes use `SUPABASE_SERVICE_ROLE_KEY` via `createAdminClient()`.
